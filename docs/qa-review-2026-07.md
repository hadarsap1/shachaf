# QA Review — Shachaf PWA
**Date:** 2026-07-02
**Scope:** `src/lib/db.js`, `src/context/AuthContext.jsx`, `src/components/layout/AppShell.jsx`, all `src/pages/**`, `vite.config.js` / service-worker config.
**Method:** Static code review (no code changes made). No test framework is exercised beyond one existing unit test (`src/lib/calendar.test.js`).

---

## Executive Summary

The codebase is generally well-structured and the team has clearly already fought (and mostly won) the "no composite index" battle — most list queries in `db.js` are commented with the reasoning and correctly sort client-side. However, this review found:

- **1 remaining latent Firestore crash** (`getResources()` uses two `orderBy()` clauses on different fields, which requires a composite index exactly like `where()+orderBy()` does — same failure class as the bugs already fixed elsewhere in the file).
- **1 confirmed JavaScript runtime bug** in `AppShell.jsx` (`ReferenceError: className is not defined`) that fires for every non-admin user without a class, in `getMemberBottomNav`.
- **A storage-orphan bug**: `deleteBusiness()` never deletes the business's Storage image, leaking files indefinitely.
- **A double-submission race** in form submission (`FormFillPage`) — no button-disable during `await`, combined with a read-then-write upsert in `saveSubmission()` that is not atomic, so a fast double-tap can create two submission docs for the same user+form.
- **Widespread missing `useEffect` cleanup** (28 of ~35 files with effects) — a systemic "set state after unmount" risk, most consequential in fast-navigating mobile use.
- **No client-side format validation** on email/phone/URL fields anywhere in the app (only "required" checks exist), and only cosmetic `type="email"/"tel"` HTML5 hints that are trivially bypassable and, worse, invisible on non-`<form>` save buttons (Business Directory, Settings) since native validation never fires without a wrapping `<form onSubmit>`.
- **localStorage schedule overrides** (`ClassPage.jsx`) silently swallow `QuotaExceededError` and have no invalidation when the class's default schedule changes — stale/lost overrides are invisible to the user.
- Zero automated tests beyond one pure-function test file; no smoke/E2E coverage exists for auth, Firestore queries, or the PWA update flow — the exact areas that have caused production incidents before.
- PWA update flow (`registerType: 'autoUpdate'` + `skipWaiting`/`clientsClaim`) has no `virtual:pwa-register` hook wired up in `main.jsx`, meaning updates activate silently but React state / open forms are not warned — mid-session users can lose in-progress form data if the SW forces a reload behavior is added later, or conversely won't be reloaded until next full navigation, risking users being stuck on a stale JS bundle referencing deleted Firestore fields/routes.

No blockers were found that are actively broken in the currently-fixed query patterns; the two blockers found (resources ordering, AppShell ReferenceError) are new/independent from the previously-fixed bugs.

## Prioritized Test-and-Fix Plan

### P0 — Fix before next deploy (both are guaranteed-reproduction crashes)
1. **`getResources()` composite-index crash** (`db.js:500`) — remove the second `orderBy()`, sort by `category` then `order` client-side, matching the pattern already used for `getChildren`, `getChildrenByParent`, `getUnlinkedChildren`, etc.
2. **`AppShell.jsx:113` ReferenceError** — `getMemberBottomNav` should accept/derive `hasClass` the same way `buildMemberNav` does (either pass `className` in as a parameter or drop the `!!className` condition, since `classIds.length > 0` already covers the meaningful case). This is currently crashing the shell for every non-admin session — verify this isn't already crash-looping in production analytics/error tracking (Sentry, etc., if configured) immediately.

### P1 — Fix this sprint
3. Fix `deleteBusiness` to fetch the doc first and delete `imagePath` from Storage (mirror the pattern already used in `deleteEventImage`/`deleteChildPhoto`/`deleteGroupFile`).
4. Add a `submitting` state to `FillView.handleSubmit` (disable submit button during `await`), and make `saveSubmission`'s upsert atomic via a deterministic doc ID (`${userId}_${formId}`) with `setDoc(..., {merge:true})` instead of query-then-write, eliminating the race entirely rather than just narrowing the window.
5. Add basic client-side format validation (regex or a small validator) for email/phone/URL fields at the two-three places they're collected (BusinessDirectoryPage, SettingsPage, OnboardingPage if applicable) before persisting to Firestore.

### P2 — Fix next sprint / opportunistic
6. Add unmounted-guards or `AbortController`-based cancellation to the highest-traffic pages first: `DashboardPage.jsx`, `ClassPage.jsx`, `EventsPage.jsx`, `TasksPage.jsx` (most-visited per the bottom-nav config), then sweep the rest.
7. Surface `saveOverrides` quota errors to the user (toast/inline error) instead of swallowing; add a lightweight schedule-version stamp (`scheduleVersion` on the class doc) so `ClassPage` can detect and clear stale overrides automatically when the admin edits the schedule.
8. Wire up `virtual:pwa-register`'s `onNeedRefresh`/`onOfflineReady` callbacks in `main.jsx` to show a "new version available — reload" toast, since `autoUpdate` + `skipWaiting` currently activates new service workers silently — users mid-form-fill can have their JS bundle swapped without warning.

### Smoke tests that would catch the most regressions (recommend Playwright or Vitest + Testing Library)
Given zero test coverage today, prioritize these as the highest-leverage smoke tests:

1. **"Every Firestore list query resolves without throwing"** — a single test that imports every exported `get*` function from `db.js` against the Firebase emulator with representative seed data (a few docs per collection) and asserts no rejection. This is the single highest-value test given the project's documented history of `FAILED_PRECONDITION` production incidents — it would have caught finding #1 and any future recurrence immediately.
2. **Render smoke test for `AppShell`** for each role (`community`, `new_family`, `host_family`, `admin`, `super_admin`, with/without `classIds`) — would have caught finding #2 instantly.
3. **Auth flow**: login → `fetchUserProfile` → correct `role`/`classIds`/`needsOnboarding` derivation, including the "new user with matching `pendingFamilies` record" and "new user with self-selected role" branches.
4. **Form submission idempotency test**: simulate two concurrent `saveSubmission` calls for the same user+form and assert exactly one document exists afterward.
5. **Storage cleanup test**: create a business with an image, delete it, assert the Storage object no longer exists (would have caught finding #3).
6. **`getChildrenByParent`/`syncUserClassIds` consistency test**: link a child, unlink a child, edit a child's `classId`, and assert the parent's `classIds` array stays in sync in each case — this directly protects against the "stale `user.classIds` vs children-derived classIds" class of bug the project has already been bitten by once.
7. **PWA update smoke test** (manual/E2E): deploy a build, load the app, deploy a second build, verify the client is prompted/updated within a reasonable time and doesn't get stuck serving stale `index.html` referencing removed chunks.

## Notes on What Is Already Solid
- The `where()+orderBy()` avoidance pattern in `db.js` is applied correctly and consistently everywhere except finding #1 — good defensive commenting throughout (e.g. lines 302, 388, 790) shows the team is aware of and actively guarding against this failure class.
- `getUsersByUids` correctly uses `Promise.allSettled` with individual `getDoc` reads rather than a batched `in` query, which is deliberate and appropriate given per-user Firestore read rules.
- File upload extension allowlisting (`BLOCKED_EXTS`/`safeExt`) is a good defense against SVG/HTML upload XSS vectors.
- `registerCoParent`/`createMember` correctly use a secondary Firebase app instance to avoid disrupting the admin's/inviter's own session — a subtle detail that's easy to get wrong and is handled well here, including cleanup in `finally`.
